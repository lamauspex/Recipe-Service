## Микросервисы

### User Service

**Назначение:** Управление пользователями и аутентификация

**Основные функции:**
- Регистрация пользователей
- Аутентификация (логин/пароль)
- Выдача JWT токенов (access + refresh)
- Валидация токенов через gRPC

**Порт:** 8000  
**База данных:** PostgreSQL (таблицы users, tokens)

**gRPC интерфейс:**
```protobuf
service UserService {
    rpc ValidateToken(TokenRequest) returns (TokenResponse);
    rpc GetUser(UserRequest) returns (UserResponse);
}
```

---

### Recipe Service

**Назначение:** Управление рецептами

**Основные функции:**
- Создание рецептов
- Валидация данных (Pydantic)
- Публикация событий в RabbitMQ

**Порт:** 8001  
**База данных:** PostgreSQL (таблицы recipes, ingredients)

**Интеграции:**
- gRPC → User Service (валидация токенов)
- RabbitMQ → События о новых рецептах

---

## Коммуникация между сервисами

### 1. Синхронная (gRPC)

```
Client → Recipe Service → User Service (gRPC)
                            ↓
                    Проверка JWT токена
                            ↓
                    Возврат user_id, email
```

**Когда используется:**
- Валидация токена при каждом запросе
- Получение данных о пользователе

### 2. Асинхронная (RabbitMQ)

```
Recipe Service → RabbitMQ → Другие сервисы/обработчики
       ↓
  Публикация события
  (recipe_created)
```

**Когда используется:**
- Создание рецепта → уведомление
- Аналитика
- Синхронизация данных

## Структура каждого сервиса

Каждый микросервис следует Clean Architecture:

```
service_xxx/
├── main.py              # Точка входа
├── src/
│   ├── api/             # Роутеры и эндпоинты
│   │   └── *.py
│   ├── service/         # Бизнес-логика
│   │   └── xxx_service.py
│   ├── repositories/    # Работа с БД
│   │   └── xxx_repository.py
│   ├── schemas/         # Pydantic модели
│   │   └── *.py
│   ├── models/          # SQLAlchemy модели
│   │   └── *.py

│   │   └── dependencies.py

└── requirements.txt     # Зависимости
```

## База данных

### PostgreSQL

Общая база данных для всех сервисов (по сервису схема):

| Таблица | Сервис | Описание |
|---------|--------|----------|
| `users` | User | Пользователи |
| `tokens` | User | Refresh токены |
| `recipes` | Recipe | Рецепты |
| `ingredients` | Recipe | Ингредиенты |

**Почему одна БД:**
- Проще настройка (один контейнер)
- ACID между сервисами
- Не требуется распределённый事务

## Безопасность

### JWT Аутентификация

```
1. Пользователь логинится → получает access + refresh токены
2. При запросе: Authorization: Bearer <access_token>
3. Recipe Service проверяет токен через gRPC → User Service
4. Если валиден → выполняется запрос
```

### Хеширование паролей

- Используется **Argon2** — современный алгоритм
- Соль генерируется автоматически
- Устойчив к GPU/ASIC атакам

## Преимущества архитектуры

1. **Независимое развёртывание** — каждый сервис можно запустить отдельно
2. **Масштабируемость** — Recipe Service можно масштабировать отдельно от User
3. **Технологическая гибкость** — каждый сервис может использовать свой стек
4. **Отказоустойчивость** — падение одного сервиса не влияет на другой
5. **Разделение ответственности** — чёткое разделение доменов
